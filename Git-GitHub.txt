## DevOps Interview – Git & GitHub

### Interviewer:

Can you explain the difference between **Git** and **GitHub**?

### Candidate:

Git is a distributed version control system used to track changes in source code locally. It allows developers to manage versions, create branches, and collaborate with others.
GitHub is a cloud-based platform that hosts Git repositories and provides collaboration features such as pull requests, code reviews, and issue tracking.

---

### Interviewer:

What is the difference between **git fetch** and **git pull**?

### Candidate:

`git fetch` downloads the latest changes from the remote repository but does not merge them into the current branch. It allows us to review the changes first.
`git pull` is a combination of `git fetch` and `git merge`. It downloads the changes and automatically merges them into the current branch.

---

### Interviewer:

How do you clone a repository?

### Candidate:

To clone a repository, we use the command:

```
git clone <repository-url>
```

This command creates a copy of the remote repository on the local machine, including all files, branches, and commit history.

---

### Interviewer:

What is the difference between **merge** and **rebase**?

### Candidate:

`git merge` combines two branches and creates a new merge commit, which keeps the full history of both branches.
`git rebase` moves or reapplies commits from one branch on top of another branch, creating a linear commit history. Merge is generally safer for shared branches, while rebase is often used to maintain a cleaner history.

---

### Interviewer:

What is a **Pull Request**?

### Candidate:

A Pull Request is a request to merge changes from one branch to another in a repository. It allows team members to review code, discuss modifications, and approve changes before they are merged into the main branch.

---

### Interviewer:

What branching strategy do you follow in your projects?

### Candidate:

In most projects, we use the **feature branching strategy**. Developers create separate feature branches from the main branch, implement changes, and push them to the repository. Then a Pull Request is created for code review, and after approval the changes are merged into the main branch.

---

### Interviewer:

What is a merge conflict and how do you resolve it?

### Candidate:

A merge conflict occurs when two branches modify the same part of a file and Git cannot automatically merge the changes.
To resolve it:

1. Run `git status` to identify conflicting files.
2. Open the file and manually resolve the conflicting code.
3. Remove conflict markers.
4. Stage the resolved file using `git add`.
5. Complete the merge using `git commit`.

---

### Interviewer:

What is cherry-picking in Git?

### Candidate:

Cherry-picking allows us to select a specific commit from one branch and apply it to another branch without merging the entire branch. It is commonly used when we want to apply a particular bug fix to another branch.

# 📘 DevOps Interview – Git & GitHub (Top 30 Questions)

---

## 🔰 Basic Level

### 1. What is **Git**?

A distributed version control system used to track changes in source code.

---

### 2. What is **GitHub**?

A cloud platform to host Git repositories and enable collaboration.

---

### 3. Difference between Git and GitHub?

Git is a tool; GitHub is a hosting platform.

---

### 4. What is a repository?

A storage location where project files and version history are kept.

---

### 5. What is a commit?

A snapshot of changes made to files.

---

### 6. What is a branch?

A separate line of development in a repository.

---

### 7. What is the default branch?

Usually **main** (previously master).

---

### 8. What is staging area?

An intermediate area where changes are prepared before committing.

---

### 9. What is `git init`?

Initializes a new Git repository.

---

### 10. What is `git clone`?

Copies a remote repository to the local system.

---

## ⚙️ Intermediate Level

### 11. Difference between `git fetch` and `git pull`?

* fetch → downloads changes
* pull → downloads + merges

---

### 12. What is `git status`?

Shows the current state of the working directory.

---

### 13. What is `git add`?

Moves changes to staging area.

---

### 14. What is `git commit`?

Saves changes with a message.

---

### 15. What is `git log`?

Shows commit history.

---

### 16. What is a **Pull Request**?

A request to merge code changes after review.

---

### 17. What is a merge conflict?

Occurs when Git cannot automatically merge changes.

---

### 18. How do you resolve merge conflicts?

* Identify file
* Edit manually
* Remove markers
* `git add` → `git commit`

---

### 19. Difference between merge and rebase?

* merge → keeps history, creates commit
* rebase → linear history, rewrites commits

---

### 20. What is cherry-picking?

Applying a specific commit from one branch to another.

---

## 🚀 Advanced Level

### 21. What is a branching strategy?

A method to manage branches during development.

---

### 22. What branching strategies have you used?

* Feature branching
* GitFlow
* Trunk-based development

---

### 23. What is GitFlow?

A structured branching model with main, develop, feature, release, and hotfix branches.

---

### 24. What is `git stash`?

Temporarily saves uncommitted changes.

---

### 25. Difference between `git reset` and `git revert`?

* reset → removes commits (history change)
* revert → creates new commit to undo changes

---

### 26. What is `git rebase` used for?

To apply commits on top of another base branch.

---

### 27. What is `git diff`?

Shows differences between files or commits.

---

### 28. What is a tag in Git?

Used to mark specific versions (like releases).

---

### 29. What is HEAD in Git?

Pointer to the current commit.

---

### 30. How do you handle large teams in Git?

* Use branching strategy
* Code reviews (PRs)
* CI/CD pipelines
* Access control in **GitHub**
